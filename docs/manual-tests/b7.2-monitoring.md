# B7.2 Monitoring & Metrics - Manual Test Script

## Overview

This document provides manual verification steps for the Prometheus metrics implementation in Agent B (B7.2).

## Prerequisites

1. Backend server running on `http://localhost:8000`
2. Database configured and running
3. `METRICS_ENABLED=true` (default in development)

## Test Cases

### 1. Metrics Endpoint Basic Access

**Development Environment (no token required):**

```powershell
# Should return 200 with Prometheus format
Invoke-WebRequest -Uri http://localhost:8000/metrics | Select-Object StatusCode, Content
```

**Expected:**
- Status code: 200
- Content contains `# HELP` and `# TYPE` markers
- Content-Type: `text/plain; version=0.0.4; charset=utf-8`

---

### 2. Metrics Endpoint Security

**Test 2a: Production without token should fail:**

```powershell
$env:ENVIRONMENT = "production"
# Restart server, then:
Invoke-WebRequest -Uri http://localhost:8000/metrics
```

**Expected:** 403 Forbidden with message about requiring token

**Test 2b: Production with valid token:**

```powershell
$env:METRICS_TOKEN = "secret"
# Restart server, then:
Invoke-WebRequest -Uri http://localhost:8000/metrics `
  -Headers @{"Authorization"="Bearer secret"}
```

**Expected:** 200 OK with metrics content

**Test 2c: Invalid token should fail:**

```powershell
Invoke-WebRequest -Uri http://localhost:8000/metrics `
  -Headers @{"Authorization"="Bearer wrong-token"}
```

**Expected:** 403 Forbidden

---

### 3. HTTP Request Metrics

**Test 3a: Verify metrics recorded for API requests:**

```powershell
# Make some API requests
Invoke-WebRequest -Uri http://localhost:8000/api/v1/sessions -Method GET
Invoke-WebRequest -Uri http://localhost:8000/api/v1/sessions `
  -Method POST `
  -ContentType "application/json" `
  -Body '{"title": "Test Session"}'

# Check metrics
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "http_requests_total"
```

**Expected:**
- `http_requests_total` includes entries for GET and POST methods
- Labels include `method`, `endpoint`, `status_code`
- Endpoint uses route template: `/api/v1/sessions` not actual path

**Test 3b: Verify endpoint labels use route templates:**

```powershell
# Make request with path parameter
Invoke-WebRequest -Uri http://localhost:8000/api/v1/sessions/abc-123-uuid

# Check metrics - should NOT contain the UUID
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "abc-123-uuid"
```

**Expected:** No matches - UUID should not appear in metrics labels

---

### 4. SSE Metrics (Separate from HTTP)

**Test 4a: Verify SSE connection tracking:**

```powershell
# In one terminal, open SSE connection (will hang waiting for events)
# Use curl or browser: GET http://localhost:8000/api/v1/sessions/{session_id}/stream

# In another terminal, check metrics
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "sse_connections"
```

**Expected:**
- `sse_connections_total` incremented
- `sse_connections_active` shows 1 while connection open
- `sse_connections_active` shows 0 after connection closed

**Test 4b: Verify SSE excluded from HTTP metrics:**

```powershell
# After making SSE request, check that /stream is NOT in http_request_duration
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "stream"
```

**Expected:** No matches in `http_request_duration_seconds` for `/stream` endpoints

---

### 5. Execution Metrics

**Test 5a: Successful execution records duration:**

```powershell
# Execute a command (requires Docker running)
Invoke-WebRequest -Uri http://localhost:8000/api/v1/exec `
  -Method POST `
  -ContentType "application/json" `
  -Body '{"command": "echo hello"}'

# Check exec metrics
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "exec_"
```

**Expected:**
- `exec_commands_total{status="success"}` incremented
- `exec_duration_seconds` has observation recorded

**Test 5b: Blocked command does NOT record duration:**

```powershell
# Try to execute blocked command
Invoke-WebRequest -Uri http://localhost:8000/api/v1/exec `
  -Method POST `
  -ContentType "application/json" `
  -Body '{"command": "rm -rf /"}'

# Check exec metrics
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "exec_"
```

**Expected:**
- `exec_commands_total{status="blocked"}` incremented
- `exec_duration_seconds` bucket counts unchanged (blocked commands don't record duration)

---

### 6. Session/Message Metrics

**Test 6a: Session creation counter:**

```powershell
# Create a session
Invoke-WebRequest -Uri http://localhost:8000/api/v1/sessions `
  -Method POST `
  -ContentType "application/json" `
  -Body '{"title": "Test"}'

# Check session metrics
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "sessions_created_total"
```

**Expected:** `sessions_created_total` incremented

**Test 6b: Message counters by role:**

```powershell
# Send a message (requires valid session_id)
Invoke-WebRequest -Uri http://localhost:8000/api/v1/sessions/{session_id}/messages `
  -Method POST `
  -ContentType "application/json" `
  -Body '{"content": "Hello"}'

# Check message metrics
(Invoke-WebRequest -Uri http://localhost:8000/metrics).Content | Select-String "messages_total"
```

**Expected:**
- `messages_total{role="user"}` incremented for submitted message
- `messages_total{role="assistant"}` incremented for response (after SSE completes)

## Verification Checklist

- [ ] `/metrics` endpoint returns Prometheus text format
- [ ] Token auth works in production mode
- [ ] HTTP metrics exclude SSE endpoints
- [ ] Endpoint labels use route templates (not raw paths)
- [ ] No banned labels (run_id, session_id, user_id) in any metric
- [ ] Blocked exec doesn't record duration
- [ ] DB pool metrics gracefully handle unavailable stats (-1)
- [ ] Single-process warning logged if WEB_CONCURRENCY > 1

## Troubleshooting

### Metrics endpoint returns 404
- Verify `METRICS_ENABLED=true` in environment
- Check server logs for startup errors

### Metrics endpoint returns 403
- In production, set `METRICS_TOKEN` and use `Authorization: Bearer <token>`
- In development, 403 should not occur unless token is set

### Missing metrics
- Verify the corresponding action was performed (session created, command executed, etc.)
- Metrics are cumulative - values persist across requests until server restart
